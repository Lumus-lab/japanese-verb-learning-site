# 日文動詞全解析系統：公開學習網站設計

## 1. 目標

將現有 React 原型整理為一個任何人都能直接開啟使用的繁體中文日文學習網站。網站專注於日文動詞變化，兼顧規則理解、常用動詞查詢與簡單練習。

第一版不需要登入、後端、資料庫、收藏或進度紀錄。使用者打開網站即可學習，關閉網站後不保留狀態。

## 2. 讀者與內容原則

- 主要讀者：以繁體中文學習日文的初學者。
- 主要顯示：日文漢字、假名與繁體中文解釋。
- 羅馬拼音：可作為次要輔助資訊，不放在主要閱讀動線中。
- 標準形優先：主畫面顯示規範形式。常見口語形式，例如一段動詞的「ら抜き言葉」，只放在備註區。
- 正確性優先：人工確認的資料與演算法推測結果必須清楚區分。

## 3. 第一版功能範圍

### 3.1 首頁

- 簡短說明網站用途。
- 提供醒目的搜尋入口。
- 接受漢字、假名與繁體中文輸入，例如 `食べる`、`たべる`、`吃`。
- 提供通往規則整理、動詞字典與快速練習的入口。

### 3.2 規則整理

以現有 `日文動詞變化規則總整理.md` 為基礎，重新整理為適合網頁閱讀的章節：

1. 三類動詞的分類方法。
2. 五段動詞、一段動詞與不規則動詞的核心變化。
3. `て形` 與 `た形` 的音便規則。
4. 完整實用變化表。
5. 常見例外與容易混淆的動詞。

教學頁中的實例可以連到對應動詞的詳細資料。

### 3.3 動詞字典

- 第一批收錄約 100 個日常生活常用動詞。
- 支援漢字、假名與繁體中文搜尋。
- 支援依動詞分類、是否為例外、辭書形字尾、主題標籤與 JLPT 參考級別篩選。
- 每個動詞有詳細頁，顯示讀音、繁中意思、分類、提示、備註與完整變化表。

### 3.4 陌生動詞推測

當日文輸入未命中已確認資料時，網站可以使用規則引擎推測變化：

- `する`、`来る` 及 `名詞 + する`：依不規則動詞處理。
- 非 `る` 結尾：依辭書形字尾推測為五段動詞。
- `る` 結尾：依前一個假名推測為五段或一段動詞。
- 無法辨識時，不產生硬湊的答案。

推測結果必須醒目標示「推測結果，可能未涵蓋例外，請再查字典」。推測資料不能進入練習題庫。

繁體中文搜尋若未命中已確認資料，只顯示尚未收錄，不進行推測。

### 3.5 快速練習

練習題只從人工確認的動詞資料產生。第一版保持精簡：

1. 判斷動詞分類。
2. 從選項中挑出指定形式的正確變化。

練習區不儲存分數、歷史紀錄或學習進度。重新整理頁面後即可重新開始。

## 4. 完整實用變化表

每個動詞需提供以下 14 種形式：

1. 辭書形
2. `ます形`
3. `ない形`
4. `て形`
5. `た形`
6. `ば形`
7. `たら形`
8. 意向形
9. 命令形
10. 禁止形
11. 可能形
12. 被動形
13. 使役形
14. 使役被動形

每個形式應有繁中名稱、簡短用途說明與產生結果。標準形式置於主欄位，口語形式只放在備註區。

## 5. 動詞資料集

### 5.1 選取原則

第一批約 100 個動詞以「日常實用」為選入標準，而不是為了湊齊 JLPT 各級數量。資料集應涵蓋：

- 日常溝通常見動詞。
- 五段、一段與不規則動詞。
- 五段動詞的不同辭書形字尾。
- 常見例外，例如 `行く`、`帰る`、`入る`、`切る`、`知る`。
- 常見的 `名詞 + する` 動詞。
- 足夠多的主題，例如移動、飲食、居家、工作、學習、人際與購物。

### 5.2 資料來源與核對

選詞與資料整理需交叉參考：

- 日本國際交流基金的生活日語教材 IRODORI，用於確認日常生活溝通場景。
- 國立國語研究所公開的 BCCWJ 詞頻資料，用於輔助檢查常見程度。
- JLPT 官方級別能力描述與常見教材，用於補充 JLPT 參考級別。
- 現有 `日文動詞變化規則總整理.md`，作為本站教學內容的起點。

新版 JLPT 沒有官方公開的逐字詞彙分級表，因此每個動詞的級別必須標示為「JLPT 參考級別」，不能宣稱為官方分級。

### 5.3 資料模型

```ts
type VerbGroup = "godan" | "ichidan" | "irregular";
type JlptLevel = "N5" | "N4" | "N3" | "N2" | "N1";
type VerbFormId =
  | "dictionary"
  | "masu"
  | "nai"
  | "te"
  | "ta"
  | "ba"
  | "tara"
  | "volitional"
  | "imperative"
  | "prohibitive"
  | "potential"
  | "passive"
  | "causative"
  | "causativePassive";

type VerbEntry = {
  id: string;
  dictionaryForm: string;
  reading: string;
  meanings: string[];
  group: VerbGroup;
  tags: string[];
  frequencyTier: "core" | "common";
  jlptLevel: JlptLevel;
  jlptLevelType: "reference";
  notes: string[];
  overrides?: Partial<Record<VerbFormId, string>>;
};
```

`id` 必須穩定且唯一。少數例外由 `overrides` 明確覆寫，例如 `行く` 的 `て形` 與 `た形`。

## 6. 技術架構

採用 React + Vite + TypeScript 的純靜態網站。沿用現有 React 原型的互動概念，但把資料、演算法與畫面拆開。

```text
src/
  components/
  data/
    verbs.ts
    forms.ts
  domain/
    conjugate.ts
    infer.ts
    search.ts
    practice.ts
  pages/
    HomePage.tsx
    GuidePage.tsx
    DictionaryPage.tsx
    LookupPage.tsx
    PracticePage.tsx
```

職責分工：

- `verbs.ts`：人工確認的常用動詞資料。
- `forms.ts`：各種變化形式的名稱、順序與用途說明。
- `conjugate.ts`：依動詞分類產生完整變化表，並套用例外覆寫。
- `infer.ts`：處理未收錄日文動詞的規則推測與警告。
- `search.ts`：處理漢字、假名與繁體中文搜尋。
- `practice.ts`：只從人工確認資料產生精簡選擇題。

## 7. 資料流與狀態

### 7.1 查詢已收錄動詞

1. 使用者輸入漢字、假名或繁體中文。
2. 搜尋模組比對人工確認資料。
3. 找到結果後，由規則引擎產生完整變化表並套用例外覆寫。
4. 詳細頁標示此結果來自已確認資料。

### 7.2 查詢陌生日文動詞

1. 搜尋未命中資料集。
2. 推測模組檢查輸入是否為可辨識的日文辭書形。
3. 若可辨識，依規則推測分類與變化。
4. 結果頁顯示明顯警告，不把結果寫回資料集。

### 7.3 練習

1. 練習模組從人工確認資料中隨機選詞。
2. 產生分類題或變化題。
3. 使用者作答後立即顯示答案與簡短說明。
4. 不保存任何狀態。

## 8. 錯誤處理

- 空白輸入：提示使用者輸入動詞或中文意思。
- 中文搜尋未命中：提示尚未收錄。
- 無法識別的陌生日文輸入：提示目前只能推測辭書形。
- 推測可能遇到 `る` 結尾例外：顯示額外警告。
- 缺少規則或資料覆寫：開發環境中拋出明確錯誤，避免靜默顯示錯誤答案。

## 9. 測試策略

### 9.1 單元測試

- 三類動詞的 14 種變化。
- 五段動詞各種字尾。
- `行く`、`帰る`、`来る`、`する` 與 `名詞 + する` 等例外。
- 漢字、假名、繁體中文搜尋。
- 推測成功、警告與拒絕推測的狀態。
- 練習題只能使用人工確認資料。

### 9.2 資料驗證

- 每筆動詞具有唯一 `id`。
- 必填欄位完整。
- `jlptLevelType` 固定為 `reference`。
- `overrides` 只能使用支援的形式。
- 每個人工確認動詞都能產生 14 種形式。
- 第一批資料筆數約為 100 筆，並涵蓋三類動詞與常見例外。

### 9.3 畫面驗證

- 桌面與手機版皆可操作。
- 搜尋、篩選、詳細頁與練習流程可正常使用。
- 已確認資料與推測結果在視覺上容易區分。

## 10. 不納入第一版

- 登入、帳號與權限。
- 後端 API 或 Cloudflare D1。
- 收藏、最近瀏覽與學習進度。
- 跨裝置同步。
- 後台編輯器。
- 每日任務、複雜統計或間隔重複系統。
- 羅馬拼音搜尋。

這些功能若未來確有需要，可以在不破壞既有資料格式的前提下逐步加入。

## 11. 版本控管與發布

1. 將本機專案整理為可建置的 React + Vite 專案。
2. 建立 GitHub repository 並 push。
3. 將 GitHub repository 連接 Cloudflare Pages。
4. 設定自訂網域。
5. 日後更新內容時，透過 commit 與 push 觸發 Cloudflare Pages 自動部署。

第一版不需要 Cloudflare Workers 或 D1。若未來加入需要伺服器狀態的功能，再評估是否引入。
